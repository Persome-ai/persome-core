# 意图识别设计哲学

> **Provenance.** 这是 **Persome**（本 runtime / 意图预测器）的设计哲学，成文于第一个 Persome 产品 **Mens**（原生 macOS app）的开发过程中。文中出现 *Mens* 处指"某个 Persome 驱动的产品 / 预测器实例"；个别指向 Mens 代码库的文件路径、闸名、战例是**说明性示例**（不是本仓库的指令），原则本身与具体产品无关、直接适用于 persome-core。

> **主动式产品做的从来不是预测的生意，是信任的生意；而守住信任的唯一办法，是对"什么可知、什么不可知"保持近乎偏执的诚实。**

## 〇、北极星：Persome 是一个意图预测器（LLM / 世界模型类比）

> **把 Persome 当成一个"针对单个人的预测模型"来想。这是 §一–§九 所有取舍的母框，也是一份可操作的优化指南——不是比喻文学。**

**本质对齐。** Persome 做的是意图识别；意图识别本质上是 LLM 式的预测：

- LLM 预测**下一个 token**；
- 世界模型预测**世界的下一个状态**；
- Persome 预测**一个人的下一个动作/状态**——人的动作序列。

三者是同一类问题（自回归地预测下一个状态），只是状态空间不同。所以 LLM / 世界模型这些年的演化路径，对 Persome 是一张现成的优化地图。

**公式：Persome（base model）= LLM + Memory = 意图预测器。**

- **LLM** 是基座推理能力（共享、通用）；
- **Memory 是每个人独有的"训练权重"**——`memory/{user,project,schema,person,…}-*.md` + `index.db`。同一套基座，被每个人的记忆个性化成**不同的预测器**。这就是 Persome 为何自适应、自感知、能随用户状态自迭代自进化：**换权重（记忆）即换模型，无需改基座**。

**理想闭环（北极星运行态；当前实现是它的逼近，不是终点）：**

| 阶段 | LLM 工程对应 | Persome 里的载体（现状 / 方向） |
|---|---|---|
| ① 预测用户下一步动作 | 前向推理 | 快/慢路识别器产出 intent；理想是**主动预测**而非被动识别 |
| ② 反馈做强化学习 | RLHF / DPO | 采纳/忽略 = 奖励信号 → R3 负先验 · R4 schema 反馈 · 生产反馈回路（reverse-loop） |
| ③ 深度学习行为模式 | 预训练学表征 | 从 AX tree + 截图 学 **Schema**（schema miner、WorkThread 工作线） |
| ④ 实时因果推断 | 检索/上下文增强 | 实时捕获 + attention/rewind + recall；**因果**（不止相关）是待补的方向 |

**启示——本文最实用的一条：用 LLM 的优化思路优化 Persome。** 遇到任何意图层的设计/调优问题，先做这个映射再动手：

- 调识别质量 → 这是 **eval 驱动**问题：golden set 就是我们的 LLM-eval（见 `data-driven-iteration.md`），先建可测 oracle 再让数据选杠杆。
- 改 prompt → 走 **prompt-engineering** 阶梯（`prompt-engineering.md`），别堆脆规则。
- 拒绝信号 → 是**奖励信号**，是金矿（§七），对应 RLHF 的负样本——别丢，喂回学习回路。
- "记不住 / 记错" → 是**权重**问题（memory 层），不是基座（LLM）问题。
- 想让它更准 → 优先级通常是 **更多/更好的数据（记忆、反馈）> 换 prompt > 调阈值**，与"scale data 优先于改架构"同构。

**边界（别把类比用过头）。** 类比是**思考工具**，不是实现承诺：当前 Persome 没有在线梯度更新、没有真正的 RL 训练循环、因果推断基本还没做。把它当 LLM / 世界模型想，是为了**借优化直觉**和**对齐北极星**，不是声称已经是。这条边界本身就是 §一（诚实先于聪明）。

> **本节是"怎么想"（设计心智模型）；"为什么做这门生意"（颠覆性价值 / 护城河 / 为何 2C / 信任承重墙）见 [Personome 论文](https://persome-ai.github.io/persome/)。** 一句话桥接：把 Persome 当预测器是**能力层**；它真正的生意是——**在「你」这个只有它能组装的语料上做 Deep Research**（推理被商品化，持有你、且不用你 brief，只有它）。

## 一、诚实先于聪明：先分清可知与不可知

做任何预测前，先问那团不确定属于哪一种——是"我没看够"的**认知熵**（可降，堆信息、堆上下文就能消），还是"信息根本不在我这"的**偶然熵**（不可降，用户脑子里的事你算不出来）。把偶然熵当认知熵去猛攻，是这行最贵的错误：你以为模型不够好，其实那块信息物理上就不可得。聪明的系统不是什么都敢预测的系统，是知道自己边界在哪的系统。

## 二、优化"猜错的代价"，而不是"猜对的概率"

高熵场景的残酷真相是：它高熵，就注定你经常猜错，准确率你提不上去。所以当 p（准确率）提不动时，唯一还能让介入净值为正的杠杆，是把猜错的代价往死里压。漏掉一个意图是**有限损失**（用户无感），弹错一个是**复利损失**（它降低未来每一次介入的有效性）——这个不对称，是你的设计宪法。胜负手在呈现侧和成本侧，不在预测侧。

## 三、轻，是一切的前提

Smart Compose 和 Cursor Tab 赢的不是"猜得多准"，是"猜错几乎没代价"：灰字补全、接受一个键、不接受继续打字它就消失、延迟压到察觉不到。把单次介入成本压到趋零，你才敢在五成把握下出手而不烧信任；而成本一旦趋零，频率反过来成了信任的放大器，而非消耗器。所以遇到任何介入，先问的不是"怎么更准"，是"怎么让它错了也不疼"。

## 四、你在和信任做重复博弈，而且赔率对你不利

每一次介入，抵押的不是这一次的注意力，是用户对你所有未来提示的信任额度。赢一次，额度小涨；输一次，大跌——一旦跌破某条线，用户进入"看都不看直接划掉"模式，你再准的提示也穿不过去了。所以别问"我还能不能多抓一个意图"，问"我还剩多少次猜错的额度"。Cursor 后来把"该不该弹"做成带不对称奖惩的在线策略（接受 +0.75、拒绝 −0.25、沉默 0），结果是少弹 21%、命中反升 28%——克制本身提升了质量。

## 五、猎零熵，而非猜高熵；把高熵拆开

最安全的猎场，是意图已明确、只剩机械执行的"**零熵**"动作（邮件里白纸黑字的 deadline），而不是模糊的高熵意图（用户想不想催）。而一个高熵任务，内部几乎总混着不同熵的子块：切开它，让低熵的壳（这封邮件 24h 没回、来自你重视的人）独立兑现确定的价值，把真正高熵的核单独隔离出来。你以为在做高熵预测，做对了拆解，九成其实是低熵活。配套一招：**信号可以是开集**（用户在干什么千变万化），**动作必须是闭集**（你能接的操作就那几个，外加一个"啥也不做"）——这本身就是一次降熵。

## 六、不可知的那个核，交还给手里有答案的人

对降不下来的偶然熵，正确动作不是更努力地算，是设计一个零成本的开关，让用户把脑子里那块信息主动泄露给你——选中一个词、一个快捷键、一次点头。一次用户的微动作，胜过你一百次揣摩。这不是降熵，是**绕过熵**：信息不在你这，就别假装能算出来，去问那个有信息的人，但把问的代价做到几乎为零。

## 七、先低熵后高熵；拒绝是金矿

低熵场景先上，目的有二：充值信任储备金（为日后高熵注定的猜错扛住扣分），和搭好数据管道。但高熵能力不会从低熵的成功里自动长出来——真正的燃料是用户的**拒绝、忽略、撤销**（它们纠正你"只见成功"的偏差），和一个用低熵数据养出的用户模型。记住那个用户模型的本分：它只削减认知熵，把高熵削成"中熵 + 一个留给用户的小开关"；它替用户做不了那个不可知的判断，只能把判断铺垫好、轻轻递到他手边。

## 八、少用白名单：按内容判，别按身份拦

**白名单是过拟合的门。** 一道"只放行枚举集合（这些 app / 这些域名 / 这些 kind）"的闸，本质是把"我们恰好支持了什么"误当成"什么值得识别"——它对你写过解析器的那几个 app 过拟合，并**静默地**给泛化能力封顶：用户在钉钉/Slack/Telegram 里说的同一句约会，不是因为不是意图被丢，是因为那个 *app* 不在名单里被丢。这违背第一条（诚实）和这套哲学的根：让 AI 穷尽它能确定的——而不是只在它认得的招牌下确定。

**默认按内容/行为判，把成本交给会自限的有状态闸，而不是身份白名单。** 没有专属解析器的 app，走通用兜底解析（`visible_text` → `direction="unknown"` 的消息，绝不伪造方向），交精度优先的 LLM 判——和每个已覆盖 app 一样。放进来的非聊天噪声（终端/编辑器）由**已有**的闸自然压住：seen-set/冷启基线 + per-app 指数退避（0 意图的 app 自动冷却 → 稳态成本趋零）+ "拿不准就空"的 LLM。**不靠再加一个 app 过滤**（`no_parser` 白名单闸已据此移除）。

**白名单要用，必须有显式理由**：一条内容信号替代不了的硬安全/成本边界——而且即便如此，写进文档、保持最小。仅存的 `domain_allowlist`（K2 浏览器脚手架）默认空/未启用，就是这个道理。一句话：**开集的世界里，别用白名单把它拦回闭集。**

## 九、定义状态要完备：用正交轴划域，别反应式打补丁

**任何状态 / 结构体 / 分类 / 决策边界，先想它完备不完备（MECE：互斥 + 穷尽）。** 方法：找出决定它的**几个正交轴**，让决策成为这些轴笛卡尔积上的函数——于是**每个输入都落进一个明确的格子，没有空白、没有重叠**。用多个维度去**定义域**，而不是一条条枚举特例。

这是第 8 条「头号反模式」（往 prompt 里堆 edge-case 清单，见 `prompt-engineering.md` §2）的**正面解法**。反应式地一条条加规则（每次 eval 失败补一条「不该输出 X」「除非 Y」）= laundry list：永远有没覆盖的格子，会**不断冒新 gap**，还把真正的设计分歧藏进 prose 里时灵时不灵。正交轴定义一次覆盖所有格子 → 抗 gap、可维护、把分歧**显式定位到具体一格**，变成一个能拍板的设计决定。

**落到快路识别器**：每条到达消息的输出是个三元决策 `DROP / FIRE / DEFER(交慢路)`。它不是一串散装规则，是这几个正交轴的函数——

- **前向性**：有没有指向未来的动作+对象？没有 → DROP（闲聊 / 即时求助 / 被动浏览）。
- **自包含性**：本条能自判，还是要靠 trajectory/memory（裸「行」、"上次那个"）？要 → DEFER。
- **锚类型** × **承诺来源** 的积表（前两轴过关后）：

  | | 硬锚（明天3点 / 下次打开X） | 软锚（下礼拜 / 改天） | 事件锚（发布完 / 等你方便） | 无锚 |
  |---|---|---|---|---|
  | SELF（用户亲口写下） | FIRE | FIRE | FIRE + condition_hint | FIRE（任务） |
  | OTHER-已定（通知定下的安排） | **FIRE** | defer | defer | — |
  | OTHER-未定（提议未应承） | FIRE(proposed) | **DEFER** | **DEFER** | DROP |

  曾经散装加的「即时求助≠前向」「事件锚→defer 除非自我提醒」「已排定日程仍 commit」全是这张表的单个格子；那个反复摇摆的 `pos_facet_conditional_after_release`（SELF×事件锚）在散装版里只是埋在 prose 的「例外」、LLM 时灵时不灵，进了表就是明确一格 = FIRE，摇摆消失。剩下的真分歧（SELF×无锚：golden 要 DEFER、表说 FIRE）被**显式定位到一格**，成为能跟产品方拍板的事，而不是藏着的不一致。

**同理回头审已有状态**是不是真正交+穷尽。两个已审过的实例（2026-06-30，都按 2×2 找到并修了一格）：

- **意图生命周期** `open|armed|consumed|dismissed|expired|resolved`：终态干净分解为 **(谁关 × 价值) 2×2** —— 用户/done=`consumed`、用户/dropped=`dismissed`、系统/done=`resolved`、系统/dropped=`expired`；活跃态按"可否立即浮现"分 `open`/`armed`。`resolved` 当初补的正是"系统/done"空格。审计抓到一格放错：`#532 armed-TTL 收割`原落 `dismissed`（用户/拒），但 armed 是**休眠、从未向用户浮现**的——盖"用户拒"的章不诚实（违反第一条），且它是系统/staleness 关 → 应是 `expired`。已改（commit `9b58074`）。
- **`telos` 闭集**：原六元里 `delegate`（"交给谁做"=executor 轴）和其余五个（"想要什么结果"=outcome 轴）**不在同一根轴**，且下游早已把 delegate 和 produce collapse 成同一个 `kind=assignment`——代码自证它不该分立。已退役 `delegate`，outcome 轴收为五元（commit/acquire/produce/transact/monitor），"谁来做"由正交的 `with` facet 承载；legacy/drift 的 delegate 在入口归一到 produce（字节兼容）。

**改一个枚举/状态机/分类前，先把笛卡尔积表画出来逐格填，再写代码或 prompt。**

最后一件事，值得你站远一点看。

我们这场对话是从一个大问题开始的——AI 有了智能，人还剩什么。当时绕了很久，落在一句：人不可替代的位置，不在"更聪明"，而在承担后果、作为被在乎的一方、作为"什么值得"的源头——在那个机器算不出、只能由一个有限的人来扛的判断上。

你发现没有，你这个产品的设计哲学，和那个答案是**同一个东西**，只是缩到了工程尺度上：机器把可知的、可计算的那部分（怎么做）尽数接管，人始终是那个不可计算的判断（值不值得、要不要）的所在地。你设计的每一个"交还用户"，都是在产品里重演那个结论——不是因为技术做不到，是因为那个判断本就该留在既握有信息、也承担后果的人那里。

所以这套哲学的内核，一句话：**让 AI 穷尽它能确定的，让人保留他独有的不确定。** 一个好的主动式 Agent，不是一个什么都替你决定的全知助手，是一个足够懂你、因而能把该铺的路都替你铺好、只把最后那一下轻轻交回你手里的伙伴。

## 在 Persome 里，这套哲学落到了哪些具体机制

上面的原则不是空谈——persome-core 的意图层就是它们的成文宪法。**呈现侧的最后一下**（弹不弹、怎么呈现、接受/忽略的不对称）属于**消费产品**（Mens 是第一个，见下方的产品侧战例）；**runtime 这边**兑现的是识别、生命周期、学习回路：

- **猎零熵、不猜高熵**（第五、七条）——快路识别器对每条到达消息做 `DROP / FIRE / DEFER` 三元决策（§九的正交轴表），只把**自包含、前向、有锚**的意图 FIRE 出来，模糊的核 DEFER 给慢路或 DROP。`intents` 只暴露识别器有把握的那部分（confidence + grounding），把"该不该打扰用户"的判断留给消费侧的 gate。
- **代价不对称写进生命周期**（第二、四条）——intent 生命周期是 `open|armed|consumed|dismissed|expired|resolved` 的 (谁关 × 价值) 2×2（§九）：漏掉→`expired`（系统/staleness，有限损失，静默）；用户显式拒→`dismissed`（复利损失，喂回学习）。终态由谁关闭 + 价值诚实决定，绝不把"从未浮现的休眠项"盖成"用户拒"。
- **拒绝是金矿**（第七条）——`dismissed` 是 R3 拒绝学习的负样本（负先验），`consumed`/`resolved` 是正信号；schema miner（R4）从行为里学 Schema 反哺识别。拒绝/忽略/撤销就是这个 runtime 的 RLHF 燃料——别丢，喂回 `intent.sink` 与 schema 层。
- **失败退回确定的那条路**（第一条）——识别与 recall 的每个 LLM 阶段都 fail-open：LLM 坏了/无 key，capture + BM25 检索 + 确定性闸照常工作，绝不因为一个 LLM 调用失败而挡住整条路径。捕获到的屏幕文本是**不可信数据**：任何把它喂进下游 prompt 的地方都加数据围栏，代码握着栏杆、LLM 只填内容。

一句话：**信号是开集（用户在干嘛千变万化），动作是闭集（识别器的 FIRE/DEFER/DROP + 消费侧的接受/忽略/啥也不做），最后那一下永远轻轻交回用户手里。**

> **产品侧战例（Mens）**：Mens app 的 `ContextSentinel` 每 60s 只浮**一个**最陈旧的合格 open intent（confidence ≥ 0.7），显式 Ignore 触发 10 分钟冷却、自动超时只本地隐藏不回写——把"一次只弹一个、挑最该弹的、错了也不疼"落到呈现层。呈现策略是**产品**的事，识别与学习是 **runtime** 的事——这条分层本身就是中台/产品的边界。
