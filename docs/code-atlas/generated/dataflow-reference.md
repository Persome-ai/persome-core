## Control plane

Owns process lifetime, policy, configuration, permissions, locks, and recovery. It changes when work may run, not the person's modeled state directly.

### `startup-recovery` — Startup integrity and recovery gate

- **Fact type:** deterministic gate
- **Why it exists:** Before an offline CLI opens mutable Runtime state or starts the daemon, resolves incomplete recovery journals and repairs or quarantines a damaged config/database under one explicit authority decision.
- **Trigger:** CLI `_init`, only when no daemon owns the Runtime lock
- **Input:** config TOML; live SQLite plus WAL/SHM; verified backups; Markdown memory; recovery journals; last model manifest
- **Output:** healthy or recovered Runtime state; forensic quarantine copy; recovery journal/marker; rebuilt retrieval projection; invalidated derived geometry when authority changed
- **Core algorithm:** Parse config and run a capped SQLite integrity check; repair purely derived FTS damage in place, otherwise journal the phase, invalidate the model manifest, quarantine the live DB, restore the newest verified backup, resolve Markdown versus evomem authority, rebuild projections, and clear regenerable geometry whose evidence source changed.
- **State / idempotency:** The pending recovery journal is a durable fail-closed gate for memory writes, model builds, and Runtime start until recovery converges or an operator resolves authority.
- **Failure semantics:** A failed real recovery keeps the journal and forensic artifacts instead of guessing between divergent authorities; live-daemon initialization does not race this offline repair path.

### `runtime-orchestration` — Daemon task registry

- **Fact type:** orchestration
- **Why it exists:** Starts exactly the configured capture, state, model, maintenance, and local-access loops and makes their enablement visible in one registry.
- **Trigger:** `persome start`, foreground daemon run, or LaunchAgent ownership
- **Input:** typed `Config`; capture-only flag; runtime generation and ownership locks
- **Output:** named asyncio tasks; owner-only runtime-state receipt; graceful or bounded shutdown
- **Core algorithm:** Build a declarative `TaskDefinition` registry, evaluate each enable predicate, create named asyncio tasks, attach crash callbacks, and cancel/drain them under a bounded shutdown deadline.
- **State / idempotency:** One daemon holds `.daemon.lock`; `.runtime-state.json` binds PID to a random generation. Task loops own their narrower locks/watermarks.
- **Failure semantics:** Unexpected task exits are logged; MCP restarts with exponential backoff; ambiguous live ownership fails closed instead of starting a second writer.

### `daily-maintenance` — Daily recovery, projection, forgetting, and survivability

- **Fact type:** orchestration
- **Why it exists:** Runs the production daily safety sequence that closes stranded work, repairs the retrieval projection, applies optional forgetting/contradiction policy, bounds operational state, and verifies recoverability.
- **Trigger:** Configured local daily safety-net time (23:55 by default), including capture-only mode
- **Input:** open/pending session state; current evo_nodes and entries projections; contradiction/decay/orphan feature gates; WAL, telemetry, backup, integrity, and projection receipts
- **Output:** force-ended/caught-up sessions; rebuilt evomem-to-entries projection; optional contradiction/decay/orphan mutations; truncated WAL and telemetry; verified backup/integrity/edit findings
- **Core algorithm:** In order: force-end the open session and run writer catch-up; rebuild entries/FTS from evomem when delta apply is active; optionally mark contradictions, decay old never-read clusters, and retire orphan Points; truncate the WAL; prune parser/retrieval telemetry; optionally create a verified retained backup and run integrity handling; finally detect manual edits under evomem authority.
- **State / idempotency:** Each mutation keeps its own receipts/status; feature gates default conservative, backups have retention, integrity may freeze writes, and failures are logged per side channel so later maintenance can continue.
- **Failure semantics:** Most tail substages fail open to the remainder of the tick and never fabricate replacement state. An exception from force-end or writer catch-up skips that day's entire maintenance tail, and there is no durable missed-run journal. Projection also runs before optional orphan retirement; these ordering/recovery boundaries are recorded as verified gaps.

### `ax-event-gate` — AX event debounce and dedup

### `privacy-pixel-gate` — Privacy and pixel gate

- **Fact type:** deterministic gate
- **Why it exists:** Skips collection behind a locked screen, strips content from secure-input observations while retaining metadata, removes structural placeholders, and encrypts or omits persisted pixels.
- **Trigger:** Before capture acquisition, during candidate finalization, and when screenshot/OCR content is attached
- **Input:** screen lock and secure-input state; raw capture candidate; AX placeholders; screenshot policy and key
- **Output:** skipped capture or metadata-only secure observation; placeholder-sanitized evidence; encrypted/omitted pixel payload
- **Core algorithm:** Fail closed before acquisition when the screen is locked; for a focused secure field remove AX, screenshot, text, URL, and OCR opportunity but retain timestamp/window metadata; sanitize confirmed structural placeholder values; encrypt screenshots with AES-256-GCM only when a valid install-generated key exists. Secret-pattern scrubbing is a separate snapshot/export boundary, not a capture-time operation.
- **State / idempotency:** Pixel policy and OCR choice are durable config; screenshot keys live only in the owner-only env file.
- **Failure semantics:** Missing/invalid screenshot key omits pixels rather than writing plaintext; a positive secure-input signal yields a persisted metadata-only record, while uncertain lock state skips the whole capture.

### `capture-commit` — Capture JSON and FTS commit

- **Fact type:** durable store
- **Why it exists:** Creates the bounded raw evidence record that every later stage can audit back to, while retaining only policy-allowed content.

- **Input:** S1 observation; content fingerprint; optional encrypted screenshot; capture timestamp

## State formation

Compresses captures into minute blocks and independently cuts work sessions. It creates bounded temporal state without creating higher-order personal geometry.

### `timeline-block` — One-minute timeline block

- **Fact type:** LLM proposal
- **Why it exists:** Normalizes a wall-clock minute of captures into a small chronological block while retaining raw focus evidence for later modeling.
- **Trigger:** Timeline tick closes a wall-clock-aligned minute not already present
- **Input:** capture JSON in `[start,end)`; OCR backfill from FTS; app parser output; timeline prompt
- **Output:** `timeline_blocks` row; normalized entries; focus excerpt/structured attention metadata; source capture IDs
- **Core algorithm:** Select at most the newest 30 captures, prioritize focused text or terminal/chat tails, render structured app evidence, ask the timeline LLM for ordered self-contained lines, validate hints, and fall back to deterministic per-app lines on failure.
- **State / idempotency:** `UNIQUE(start_time,end_time)` plus `INSERT OR IGNORE` makes the stored block idempotent; block carries source capture IDs and parser telemetry.
- **Failure semantics:** Malformed/failed LLM output degrades to heuristic entries. Concurrent producers can both spend the timeline LLM call before only one row wins the unique insert.

- **Why it exists:** Compresses the strongest active personal structure into at most one apex without erasing a previously valid Root when evidence is insufficient.
- **Trigger:** Root stage of an enabled structural build
- **Input:** active Faces and Volumes; profile evidence; token budget; root prompt

- **State / idempotency:** Exactly zero or one live level-3 row; prior versions and aggregated receipts remain auditable.
- **Failure semantics:** No evidence, provider error, or invalid response returns a degraded reason and never replaces a valid Root with empty content.

### `vector-index` — Optional vector backfill

- **Fact type:** read-only projection
- **Why it exists:** Adds an optional local similarity projection over accepted live entries without becoming a new write authority.
- **Trigger:** Sixty-second embed tick or final stage of model build when hybrid retrieval is configured
- **Input:** pending live entry IDs/text; configured embeddings endpoint/model; batch limits
- **Output:** packed float vectors; queue state; cached live matrix
- **Core algorithm:** Enqueue accepted entries, batch the provider call, validate response cardinality, pack each float array with its recorded dimension into SQLite, garbage-collect orphan vectors, and cache a validity-token keyed live matrix while excluding incompatible dimensions at read time.

## Projection and retrieval

Builds read-only, versioned views over current and historical model state and retrieves evidence without creating a second model.

### `snapshot-projection` — Versioned model snapshot

- **Fact type:** read-only projection
- **Why it exists:** Projects the current/historical Point, Line, Face, Volume, Root, receipt, and build state into one validated public contract without mutating it.
- **Trigger:** Model graph/status/export request or build completion
- **Input:** all non-archived evomem nodes, including history/shadow; supersede and active/open relation edges; active schemas; valid build manifest; redaction policy
- **Output:** schema-versioned snapshot; build/stats object; redacted owner-only export
- **Core algorithm:** Query all non-archived evolution Points, derive supersedes Lines, include only active/open relation Lines and active geometry, close receipts through membership, enforce at-most-one Root and manifest invariants, redact exports deterministically, and write them atomically mode 0600.
- **State / idempotency:** Snapshot is read-only and reproducible for fixed data/time/manifest; HTTP/MCP/CLI share the same contract.
- **Failure semantics:** Contract inconsistency raises instead of guessing; absent/invalid completed manifest reports `not_built`; export defaults to redaction.

### `associative-retrieval` — Hybrid associative retrieval

- **Fact type:** read-only projection
- **Why it exists:** Retrieves current evidence across lexical, temporal, relation, and optional vector cues and can expand it into auditable chains.
- **Trigger:** MCP/CLI/API memory or context query
- **Input:** query distilled into time/scene/term slots; FTS BM25 pools; relation/window pools; optional dense vectors
- **Output:** RRF-fused and MMR/breadth-cut entry hits; retrieval counters; evidence chains
- **Core algorithm:** Distill a multi-slot query, run weighted lexical/contains/time/relation/dense pools, fuse ranks with reciprocal-rank fusion, apply recency and optional character-bigram MMR, then reconstruct bounded oriented relation-edge chains to self.

## Local access and correction

Exposes owner-local CLI, REST, viewer, and MCP surfaces. Read paths project state; explicit write paths remain authenticated and audited.

### `local-read-surfaces` — REST, viewer, MCP, and CLI reads

- **Fact type:** access surface
- **Why it exists:** Makes the same owner-local runtime/model facts available to humans and agent clients without introducing product dashboards or a second model.
- **Trigger:** Authenticated owner-local HTTP/MCP request or local CLI command
- **Input:** snapshot/retrieval projection; owner bearer or one-use viewer capability; bounded request arguments
- **Output:** REST JSON; local Three.js `/model`; MCP tool response; CLI text/JSON
- **Core algorithm:** Protect every route except canonical health with bearer/capability middleware, validate body/tool limits, call shared snapshot/retrieval functions, render packaged local viewer assets, and watchdog MCP parent/process lifetime.
- **State / idempotency:** Loopback is the default bind; bearer-authenticated POST issues a one-use nonce, whose bearer-exempt GET exchange creates a path-scoped HttpOnly browser session; no model content is uploaded by the viewer.
- **Failure semantics:** Unsafe bind, missing/invalid bearer or capability, oversized input, or absent build fails explicitly. Canonical health and the nonce-protected bootstrap GET are the only bearer-exempt routes.

### `explicit-correction` — Explicit audited correction

- **Fact type:** explicit audited write
- **Why it exists:** Lets the owner correct, supersede, retype, merge, reject an alias, or revoke state through a narrow auditable entrance.
- **Trigger:** Explicit CLI/MCP correction request
- **Input:** bounded user signal; candidate search results; requested operation; local authorization
- **Output:** supersede/retype/merge/revoke operation; audit log; targeted Face re-mine and Root re-synthesis
- **Core algorithm:** Bound and authenticate the request, retrieve a small candidate set, plan an allowed operation, execute through deterministic store/evomem entrances, log source/reason/applied IDs, then targeted-remine affected Faces and immediately resynthesize Root.

- **Failure semantics:** Ambiguous or invalid operations are rejected rather than free-form editing; a corrected Face can temporarily coexist with stale Volumes, and the immediately resynthesized Root can therefore still consume old Volume state.

### `agent-remember` — Agent finding write-back

- **Fact type:** explicit audited write
- **Why it exists:** Appends an authorized agent finding to one dedicated durable memory file with immutable source/run provenance, without invoking the correction planner or structural re-forward path.

- **Input:** non-empty bounded content; optional caller tags; optional agent run ID

- **Failure semantics:** Empty content is rejected; integrity or storage failures propagate and no free-form correction, Face re-mine, or Root synthesis is attempted.

### `explicit-secure-erasure` — Explicit secure erasure

- **Fact type:** explicit audited write
- **Why it exists:** Deletes the operator-selected capture, timeline, memory, or entire local Runtime scope across live, backup, quarantine, export, Markdown, and SQLite/FTS copies instead of deleting only the visible primary file.
- **Trigger:** Confirmed `persome clean captures|timeline|memory|all` while the daemon is stopped
- **Input:** selected deletion scope; live database and sidecars; capture/memory files; backups and quarantines; exports and recovery artifacts
- **Output:** securely purged scope; vacuumed/truncated SQLite and FTS; removed derived snapshots/markers; explicit failure receipt when any retained copy cannot be scrubbed
- **Core algorithm:** Require stopped-daemon ownership and confirmation, remove canonical files and rows, scrub matching tables from backups/quarantines, enable SQLite/FTS secure deletion, rebuild indexes, truncate WAL, VACUUM, and delete derived model/export/recovery artifacts for the chosen scope.
- **State / idempotency:** Erasure is an operator-only destructive boundary; scope-specific helpers count removed artifacts and never reuse normal retention as proof of secure purge.
- **Failure semantics:** Busy WAL, unsafe retained copies, or scrub/remove failures fail loudly; the command does not claim success while a known local copy still contains the selected data.
